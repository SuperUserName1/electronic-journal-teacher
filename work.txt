Техническое задание №2024-ЭЖП
на разработку программы для ЭВМ "Электронный журнал Преподавателя"

1. Введение
1.1. Наименование программы
Наименование программы: "ЭЖП: Электронный Журнал Преподавателя" (рабочее название).
1.2. Краткая характеристика области применения
Программа предназначена для автоматизации рабочего процесса преподавателей высших учебных заведений. Она позволяет систематизировать учет посещаемости, успеваемости, учебных материалов и домашних заданий, а также предоставляет инструменты для аналитики и прогнозирования успеваемости студентов.

2. Основания для разработки
Разработка осуществляется на основании потребности в цифровизации учебного процесса, повышения эффективности работы преподавателя и объективного контроля успеваемости студентов.

3. Назначение и цели разработки
3.1. Назначение
Создание интуитивно понятного, кроссплатформенного desktop-приложения, которое предоставит преподавателю единое пространство для управления всеми аспектами учебной деятельности по одной или нескольким учебным группам и дисциплинам.
3.2. Цели
    • Автоматизация учета посещаемости и оценок.
    • Централизованное хранение информации о пройденных темах и домашних заданиях.
    • Исключение ошибок ручного расчета среднего балла.
    • Предоставление преподавателю инструмента для прогнозирования рисков неуспеваемости студентов.
    • Повышение прозрачности учебного процесса.

4. Требования к программе
4.1. Требования к функциональным характеристикам
Модуль 1: Управление данными
    • Ф1.1. Импорт студентов: Загрузка списка студентов из внешнего файла (форматы: .csv, .xlsx). Обязательные поля: ФИО, группа, номер зачетной книжки.
    • Ф1.2. Ручное редактирование студентов: Возможность добавлять, редактировать и удалять записи о студентах вручную.
    • Ф1.3. Создание и управление курсами/дисциплинами: Преподаватель может создать новую дисциплину, привязать ее к академическому семестру и назначить на нее одну или несколько учебных групп.
Модуль 2: Работа с занятиями
    • Ф2.1. Ведение календаря занятий: Возможность вручную добавлять даты проведенных занятий.
    • Ф2.2. Запись темы занятия: Для каждой даты должна быть возможность указать тему пройденного урока.
    • Ф2.3. Назначение ДЗ: Для каждого занятия можно указать текст домашнего задания и срок сдачи.
Модуль 3: Учет успеваемости и посещаемости
    • Ф3.1. Ввод оценок: Для каждого студента и каждой даты занятия предусмотреть поле для ввода оценки (по произвольной шкале, например, 5-балльной, 10-балльной, 100-балльной).
    • Ф3.2. Учет посещаемости: Для каждого студента и каждой даты занятия предусмотреть индикатор посещения (например, галочка "присутствовал", "н" - отсутствие, "у" - уважительная причина).
    • Ф3.3. Взвешивание оценок: Возможность назначить "вес" для каждого типа работ (например, лекция - вес 1, практика - вес 1.5, домашняя работа - вес 2, экзамен - вес 3).
Модуль 4: Расчеты и аналитика (Ключевой модуль)
    • Ф4.1. Автоматический расчет среднего балла:
        ◦ Расчет текущего среднего балла для каждого студента по каждой дисциплине с учетом весов работ.
        ◦ Отображение среднего балла в реальном времени.
    • Ф4.2. Прогнозирование итоговой оценки и рисков:
        ◦ Алгоритм прогноза: На основе текущего среднего балла, динамики изменения оценок (улучшение/ухудшение) и процента посещаемости программа должна рассчитывать вероятную итоговую оценку.
        ◦ Визуализация рисков: Студенты должны быть помечены цветовыми индикаторами (например, "зеленый" - нет рисков, "желтый" - риск сдачи на низкий балл, "красный" - риск неудовлетворительной сдачи).
        ◦ Факторы риска: Низкий средний балл, негативная динамика, высокий процент пропусков.
Модуль 5: Отчетность и экспорт
    • Ф5.1. Просмотр ведомости: Формирование таблицы успеваемости по группе с возможностью фильтрации по дисциплине и периоду.
    • Ф5.2. Экспорт данных: Возможность экспорта итоговых ведомостей в файлы формата PDF или XLSX.
Модуль 6: Системные функции
    • Ф6.1. Резервное копирование: Возможность создания резервной копии базы данных.
    • Ф6.2. Восстановление из резервной копии.
4.2. Требования к надежности
    • Программа должна обеспечивать целостность данных при некорректном завершении работы.
    • Автоматическое сохранение вводимых данных с интервалом не более 1 минуты.
4.3. Требования к интерфейсу (UI/UX)
    • Тип: Графический пользовательский интерфейс (GUI).
    • Стиль: Минималистичный, интуитивно понятный. Основная навигация - через боковое меню или вкладки.
    • Основной экран: Должен представлять собой таблицу (журнал) с датами занятий по горизонтали и списком студентов по вертикали. Ячейки на пересечении содержат оценки и отметки о посещении.
    • Кроссплатформенность: Предпочтительная разработка на фреймворке, позволяющем запускать приложение на ОС Windows, Linux, macOS (например, Electron, Qt, JavaFX).
4.4. Технические требования
    • Язык программирования: Python, JavaScript, C# или другой современный язык.
    • Хранение данных: Для standalone-версии - встроенная база данных (SQLite). Для сетевой версии - клиент-серверная архитектура с СУБД (PostgreSQL/MySQL).
    • Локализация: Интерфейс на русском языке.

5. Стадии и этапы разработки
    1. Стадия 1: Проектирование
        ◦ Создание прототипов интерфейса.
        ◦ Проектирование структуры базы данных.
        ◦ Согласование ТЗ с Заказчиком.
    2. Стадия 2: Разработка
        ◦ Реализация модуля управления данными (Модуль 1).
        ◦ Реализация модуля работы с занятиями (Модуль 2).
        ◦ Реализация модуля учета успеваемости (Модуль 3).
        ◦ Реализация модуля расчетов и аналитики (Модуль 4) - ключевой этап.
        ◦ Реализация модуля отчетности (Модуль 5).
    3. Стадия 3: Тестирование и отладка
        ◦ Модульное тестирование.
        ◦ Интеграционное тестирование.
        ◦ Бета-тестирование с привлечением конечных пользователей (преподавателей).
    4. Стадия 4: Ввод в действие
        ◦ Подготовка установочного пакета.
        ◦ Написание пользовательской документации.
        ◦ Передача программы и документации Заказчику.

6. Порядок контроля и приемки
    • Периодичность отчетности: Разработчик предоставляет еженедельные отчеты о ходе работ.
    • Приемочные испытания: Проводятся на специально подготовленном наборе тестовых данных, предоставленных Заказчиком.
    • Критерии приемки:
        1. Реализованы все функциональные требования, указанные в п. 4.1.
        2. Интерфейс соответствует утвержденным прототипам.
        3. Программа стабильно работает на целевых платформах.
        4. Успешно пройдены все виды тестирования.
        5. Предоставлена полная пользовательская документация.

Приложение: Описание алгоритма прогнозирования
Цель: Предсказать итоговую оценку и оценить риск неуспеваемости для каждого студента.
Входные данные:
    • S - Текущий средний балл студента (с учетом весов).
    • T - Динамика оценок (тренд). Рассчитывается как линейная регрессия по всем оценкам студента за время курса. Отрицательный наклон - ухудшение, положительный - улучшение.
    • A - Процент посещаемости (количество посещенных занятий / общее количество занятий).
    • W - "Вес" оставшихся контрольных точек (экзамен, зачет, итоговый проект).
Алгоритм:
    1. Расчет прогнозируемого балла (P):
        ◦ P = S + (T * k) , где k - эмпирический коэффициент, усиливающий влияние тренда на прогноз (например, количество оставшихся недель).
        ◦ ИЛИ более сложная формула: P = (S * Completed_Weight) + (Expected_Future_Score * W), где Expected_Future_Score оценивается на основе T.
    2. Оценка уровня риска (R):
        ◦ Создается комплексный показатель риска: Risk_Score = f(S, T, A).
        ◦ Например:
            ▪ Если P < min_pass_score (минимальный проходной балл), то риск = "КРИТИЧЕСКИЙ".
            ▪ Если A < 70%, то риск повышается на один уровень.
            ▪ Если T < -0.1 (заметное ухудшение), то риск повышается на один уровень.
Визуализация в интерфейсе:
    • Зеленая зона: P >= 4.0 (по 5-балльной шкале), Risk_Score низкий.
    • Желтая зона: P между 3.0 и 4.0, Risk_Score средний. Требуется внимание.
    • Красная зона: P < 3.0, Risk_Score высокий. Необходимо срочное вмешательство.
